你给一个 Agent 派了活:盯部署、盯收件箱、盯行情。于是它每隔几小时汇报一次:*"一切正常,无事发生。"*

前十次你会看。到第一百次,你的目光已经从它身上滑过去了。而在它终于有话要说的那一天,你连那条也一起滑过去了——因为你已经被**训练**成了这样,一条"一切正常"训练一次,把这个发信人归类成噪声。

这不是 Agent 的问题,是**注意力**的问题。而且 Agent 那边能力越强,问题越糟:十个各自很有帮助的 Agent,合起来就是对一个人前额叶皮层的一次拒绝服务攻击。

我做了 **[Chief](https://github.com/SmileLikeYe/agent-chief)** 来当这块前额叶——一个本地优先的层,坐在你和所有争夺你注意力的东西之间,自己思考,然后对每个事件只做五件事之一:**打断 · 摘要 · 派发 · 收藏 · 丢弃**。

这篇文章不讲产品,讲让它不至于沦为 `if urgent: notify()` 玩具的三个工程决策:**先做便宜的思考、给每一次昂贵的思考算钱、以及证明这东西真的会学。** 后两个,恰好是我在面试里被反复追问的东西。

## 一、大多数事件不配用 LLM

最朴素的设计是把每个事件都丢给模型问一句"该不该打断?"。这慢、贵,而且——对那波"一切正常"的洪水来说——是对模型的侮辱。第 400 条一模一样的心跳,不需要一个前沿大模型来裁决。

所以 Chief 是一个漏斗,最便宜的一级先上:

![三级漏斗:24 个事件流入,只有 25% 到达 LLM](/assets/blog/chief-funnel.svg)

- **一级——硬规则(微秒)**。静音主题、去重、安静时段,以及一个*零信息*检测器,专杀空报告。(这个检测器要求**同时**命中正则**和**与一个预置"空报告"集合的嵌入相似度——所以一份只是*提到*"all clear"的安全扫描依然能通过。便宜,但不蠢。)
- **二级——相似度分类器(毫秒)**。像你以前忽略过的 → 丢弃;像你以前处理过的 → 按历史路由。还是没有 LLM。
- **三级——LLM 裁判**,只给真正新颖的事件,给五个维度打分(紧急度、相关性、可行动性、新颖性、置信度)。

跑一遍 Chief 自带的确定性"工程师的一天"回放,漏斗的价值就出来了:

```
24 个事件流入 → 只打断 1 次
96% 被拦截 · 只有 25% 的事件到达 LLM
```

四分之三的流量在花掉一个 token 之前,就在微秒级免费解决了。这不是后来补上的优化——这是整个系统的形状。

## 二、如果一次判断要花钱,那就把钱算清楚

一旦三级涉及付费 API,"该不该打断?"就有了一个美元数字。大多数系统从不看它。Chief 把它标在每一个决策上。

有两个工程细节,让这件事从装饰变成了真事:

**提示词缓存感知的定价。** 提示词按 `[system][context][user]` 排布——稳定前缀在前——所以 system 和 context 块在当天第一次之后,每次调用都是缓存*命中*。按 DeepSeek 的公开价目,缓存命中比未命中便宜约 4 倍,所以这个排布不是化妆,它就是成本的大头。Chief 显式建模了这个拆分:

```
70% 的裁判输入 token 是缓存命中 → 每 1,000 个事件约 $0.10
```

**按模型计价,而不是按厂商。** 这一条是我踩进去的坑,也是一次对抗式 review 抓出来的:一个按*后端*键控的价目表,会把 `gpt-4o-mini` 的配置按 `gpt-4o` 的费率计费——**高估约 17 倍**。修法是一个按有序子串匹配的模型级价目表(真实的 model id 长这样:`claude-3-5-haiku-20241022`,所以锚定前缀会漏)。一个悄悄差 17 倍的成本核算,比没有还糟。

于是每个决策都带着自己完整的收据,事后可回放:

```console
$ chief trace evt_20260706_1040_ab12
route dispatch at stage 3 in scene deep_work (confidence 0.85)
score 0.87  urgency=0.90 relevance=0.90 actionability=0.85 ...
┌────────────┬───────┬──────────────────────┐
│ stage      │    ms │ note                 │
│ stage1     │   0.1 │ no hard rule fired   │
│ judge      │ 812.4 │ backend deepseek     │
│ route      │   0.3 │ routed dispatch      │
└────────────┴───────┴──────────────────────┘
tokens: 1104 in (704 cached) / 96 out · cost $0.000301
```

没有黑箱。每条路由都有理由、五个分项得分、命中的规则,以及它花了多少钱。

## 三、"它会学你的偏好"是一句主张——所以我把它做成可证伪的

这个赛道里每个产品都说自己"会学"。几乎没有一个把学习过程摆给你看。信号其实就在那儿:Chief 打断你时,你可以点 **👍 值得** 或 **👎 别烦我**。那是一个奖励(reward)。按主题的权重是策略(policy)。一次 EMA 更新是训练(training)。这个闭环是架构隐含的——但它真的*闭合*吗?

所以我做了一个专门去**证伪**它的评测。一个模拟用户有隐藏的、按主题的偏好。Chief 从瞎猜开始(权重均匀),**只**被 ±1 信号纠正——没有标注、没有梯度。我们逐轮测量路由与用户真实偏好的一致率:

![学习曲线:一致率从 0% 爬到 100%,两轮收敛](/assets/blog/chief-learning-curve.svg)

```console
$ chief eval --learning
Routing agreement: 0% → 100% (+100%) · converged in 2 rounds
```

这条曲线是**构造上单调**的——一个主题一旦路由正确,反馈就停止,于是学到的偏好不会回退——而且它是确定性的,所以文档里写的数字就是你会跑出来的数字。

有意思的是这个评测**拒绝声称**什么。因为权重有界,单靠偏好无法把一个在每个维度上都近乎零信号的事件推过打断线。我本可以靠精选测试把这一点藏起来。相反,这个 harness 把这个天花板写进了报告,还有一个专门的测试把它钉死。反馈移动的是*边界*上的判断——而这恰恰是它该干的活,因为明摆着的事情已经被一级规则和清晰的高/低分处理掉了。**能说清楚你的系统在哪里失效,比声称它从不失效更有说服力。**

## 我没预料到最重要的一环:像对手一样审查它

我用一群 review Agent 来构建 Chief,让它们从相互独立的角度攻击每一次改动——逐行、删除行为、跨文件、安全、约定——然后再来一个独立的验证 pass,只保留它能从代码里**构造出来**的发现。它找到了单趟 review 会漏的真东西:控制台里一条能外泄本地 API token 的存储型 XSS 路径、上面那个 17 倍的计价 bug、一个存在会被 webhook 覆写的命名空间里的降级标记。

我学到的不是"AI 能找 bug"。而是:**带怀疑验证器的、对抗式的、多视角的 review,抓到的是和测试不同、也更吓人的一类 bug**——那种代码在精确地做你写的事、而你写的东西本身就是问题的 bug。

## 落到哪儿

Chief 现在是约 300 个离线测试、五条路由、一个三级漏斗、按模型的成本核算、一个可证伪的学习闭环,以及一个本地控制台——全部本地优先:一个 SQLite 文件加几份 markdown,无云端、无遥测。

但这三个想法能走得比这一个仓库更远:

1. **按决策值多少钱,给你的算力分级。** 别把第 400 条心跳送给前沿模型。
2. **给每一次昂贵的思考按模型、缓存感知地算钱——否则就别声称你在乎成本。**
3. **如果你说你的系统会学,那就把那个能证明它没学会的评测一起交付。**

60 秒试一下,零密钥、完全离线:

```bash
uvx agent-chief demo
```
