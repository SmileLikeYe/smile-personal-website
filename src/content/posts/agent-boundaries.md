过去一年我写了不少 Agent 工作流，最大的心得反直觉：**Agent 产品的质量，取决于你不让模型做什么。**

## 模型擅长的和代码擅长的

模型擅长的是理解模糊输入、在开放空间里做判断；代码擅长的是精确、可重复、可审计。很多 Agent 产品的失败模式，是把本该用代码写死的环节交给了模型自由发挥：

- **循环控制**交给模型 → 模型会"忘记"自己跑到第几步，或者提前宣布完成；
- **结果校验**交给模型 → 模型给自己打分永远偏高，必须用独立的验证器；
- **副作用操作**（发消息、删文件、花钱）交给模型直接执行 → 出一次事故就足够让用户永远卸载。

我现在的分界线很简单：**判断交给模型，流程和副作用留在代码里。** 模型负责"该做什么"，harness 负责"怎么做、做几次、失败了怎么办"。

## 校验是 Agent 的一半

一个可靠的 Agent 循环长这样：计划 → 执行 → **独立校验** → 修复 → 交付。中间那步独立校验，工作量大概占整个系统的一半，但它决定了产品是"演示级"还是"生产级"。

校验的形式取决于领域：写代码就跑测试和构建，改 UI 就截图对比，做数据就抽样核对。共同点是：**校验信号必须来自模型之外**——编译器、测试框架、真实渲染结果，而不是模型自己的复述。

## 边界感是产品能力

把这个思路推广开：Agent 时代的产品经理和工程师，核心竞争力是为一个模糊任务画出"模型区"和"代码区"的边界。边界画对了，一个中等模型也能给出稳定的产品体验；画错了，最强的模型也只是一个更贵的随机数生成器。

这也是我看好"harness 工程"这个方向的原因：模型能力在涨，但边界设计的价值不会消失，只会往更高的抽象层移动。
