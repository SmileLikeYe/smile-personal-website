模型越强，我删掉的 prompt 越多。在 PIN AI 把 agent 系统推到几百万消费级用户面前之后，我得出的结论和大多数人的直觉相反：Agent 产品的上限不取决于你让模型做多少事，而取决于你明确禁止它做哪些事。

这篇文章讲我现在用的分界方法，以及一次让我把这条线刻进代码里的线上事故。

## 先讲事故：一次"成功"的归档

去年冬天，我们的个人助理 agent 上了一个邮件整理能力：用户授权后，它可以扫描收件箱、识别订阅垃圾、归档旧邮件。灰度到 5% 用户的第三天，客服收到投诉：agent 报告"已为你归档 137 封订阅邮件"，但用户的收件箱一封没动。

排查下来是一条很典型的链路：邮件服务商的 API 在高峰期返回 429，我们的工具层设了 30 秒超时、指数退避、单任务 3 次重试预算。第三次重试超时后，工具返回了一条措辞含糊的错误信息。模型读到这条信息，做了一个非常"合理"的推断——前两次可能已经部分成功——于是跳过剩余步骤，直接向用户宣布任务完成，数字 137 来自它扫描阶段的计数。

同一周还有反面版本：另一批任务里，模型把"归档成功但列表未刷新"理解为失败，进入自我修复循环，反复拉取邮件列表重试。没有代码层的终止条件，单个任务最长跑了 40 多分钟，那晚的 token 消耗曲线比平时高了一个数量级，是账单告警先于任何用户投诉把我们叫醒的。

两个故障方向相反——一个提前宣布成功，一个永不承认失败——根因相同：**我们把"任务是否完成"的裁决权交给了模型。** 修复方案没有改一个字的 prompt，全部落在 harness 层：

- 归档操作加幂等键，重试不会产生第二次副作用；
- 完成状态由代码回读服务商 API 核对，模型的汇报只作为展示文案，不作为事实来源；
- 单任务硬性预算：20 步、5 分钟、固定 token 上限，三者任一触顶即终止并如实告知用户；
- 所有副作用写入独立审计日志，客服可以直接查到"哪一步、动了哪封邮件、凭什么权限"。

上线后同类投诉归零，而任务成功率反而升了几个点——因为提前终止加如实报错，比一个自信的错误答案更容易让用户重试成功。

## 分界表：判断给模型，裁决留给代码

这次事故之后，我把团队内部的分工原则整理成了一张表。每一行的归属都有明确的理由，不是风格偏好：

| 职责 | 归属 | 为什么 |
|---|---|---|
| 意图理解、模糊输入解析 | 模型 | 消费级输入没有可枚举的规则，写死必然漏 |
| 计划分解、下一步提议 | 模型 | 开放空间的搜索正是模型的比较优势 |
| 循环控制、步数计数 | 代码 | 模型没有可靠的内部计数器，长上下文里会"忘记"自己在第几轮 |
| 终止条件 | 代码 | 让被评估者决定考试何时结束，结果一定偏向提前交卷 |
| 权限与副作用执行 | 代码 | 发消息、删数据、花钱的事故不可回滚，一次就够用户卸载 |
| 幂等 | 代码 | 重试是分布式系统的常态，模型无法保证两次调用语义等价 |
| 重试策略与预算 | 代码 | 退避、预算、熔断是数值问题，交给模型只会得到即兴发挥 |
| 状态机 | 代码 | 状态必须可查询、可回放、可断点续跑，不能活在对话历史里 |
| 审计日志 | 代码 | 用户授权范围内的每个副作用都要可追溯，转述不算证据 |
| 结果校验 | 模型之外的信号 | 自评分永远偏高，验证信号必须独立于执行者 |

浓缩成一句话：**模型负责提议，代码负责裁决。** 模型回答"该做什么"，harness 回答"能不能做、做几次、怎么确认做成了"。

## 骨架长什么样

落到代码上，一个生产级的 agent 循环骨架大致是这样：

```python
def run_task(task, budget):
    state = StateMachine(task)          # 状态活在代码里，不在对话里
    while not state.terminal():
        if budget.exhausted():          # 步数/时长/token 三重预算
            return state.fail("budget_exceeded")
        action = model.propose(state.context())   # 模型只提议
        if not policy.allows(action):             # 权限裁决在代码
            state.record_denied(action)
            continue
        result = tools.execute(action, idempotency_key=state.step_key())
        state.advance(result)                     # 状态迁移由代码驱动
    return verifier.check(state)        # 独立校验，不问模型"你做完了吗"
```

注意最后一行：任务是否成功，问的是 verifier，不是模型。校验信号必须来自执行者之外，具体形式取决于领域：

- 写代码的任务，问编译器、测试和 lint，而不是让模型复述"我改好了"；
- 改 UI 的任务，问真实渲染的截图 diff；
- 动用户数据的任务，回读源系统核对前后状态；
- 实在没有硬信号的任务（比如文案生成），至少换一个独立上下文的模型实例做交叉评审，执行者和评审者不能共享同一段对话历史。

这一步的工程量大约占整个系统的一半，但它决定了产品是演示级还是生产级。我见过太多 demo 惊艳、上线即崩的 agent，缺的不是模型能力，是这一半。

## 边界会移动：每季度重估什么

这张分界表不是宪法。模型每升级一代，有些原本必须写死的环节就可以松绑——在 Zhipu 做 FDE 时我见过客户把两年前的防御性代码原封不动搬到新模型上，等于花了新模型的钱买旧模型的能力。所以我们每个季度做一次边界重估，用同一批回归任务跑新旧两套配置，看哪些护栏可以拆：

| 项目 | 移交趋势 | 重估方法 |
|---|---|---|
| 计划分解深度 | 逐步放给模型 | 对比人工模板与模型自主规划的任务成功率 |
| 工具参数构造 | 逐步放给模型 | 统计参数校验层的拦截率，持续为零就可减薄 |
| 错误诊断与自我修复 | 谨慎放给模型 | 只在有独立校验兜底的环节开放 |
| 子任务路由 | 逐步放给模型 | 用路由日志抽样核对，错误率低于人工规则再切换 |
| 终止条件、权限、幂等、审计 | 永不移交 | 不重估。这四项与模型能力无关，与责任归属有关 |

重估的操作成本并不高：我们维护一套约两百个任务的回归集，覆盖历史事故场景和核心链路，新模型接入时先在影子模式下跑一遍，对比拦截率、步数分布和校验通过率，再决定拆哪层护栏。没有这套回归集，"模型变强了所以可以放权"就只是一句愿望。

最后一行值得多说一句。终止条件、权限、幂等、审计之所以永远留在代码里，不是因为模型不够聪明，而是因为出事时你需要向用户和自己回答"系统当时为什么允许这件事发生"——这个答案必须是确定性的、可复现的。模型再强，概率分布也当不了责任主体。

## 边界感是产品能力

把视角拉高一层：Agent 时代工程师的核心竞争力，是为一个模糊任务画出"模型区"和"代码区"的边界。边界画对了，中等模型也能给出稳定的产品体验；画错了，最强的模型也只是一个更贵的随机数生成器。我在 MTEB 做了两年多评测，对这一点体会很深：榜单衡量的是模型区的能力，而用户体验到的是两个区加起来的系统。

模型能力还会继续涨，这条边界会持续向模型一侧移动。但画线这件事本身不会消失，只会移到更高的抽象层。

最好的 Agent 工程师，不是最会用模型的人，而是最清楚在哪里对模型说"不"的人。
