大多数团队相信 eval 的问题是"没有"。在 MTEB 社区做了两年多贡献之后，我的看法相反：**更危险的不是没有 eval，而是有一套你不再怀疑的 eval。** 没有评测的团队至少知道自己在抛硬币；拿着一套腐坏评测集的团队，会理直气壮地把产品往沟里带。

这篇文章分两半：前一半讲 benchmark 是怎么坏掉的——这是我在 MTEB 里反复看到的事；后一半讲我在 PIN AI 搭产品评测体系时，怎么把这些教训翻译成工程决策。

## Benchmark 的三种死法

MTEB 是文本嵌入领域事实上的标准榜单。也正因为它是标准，它承受的攻击面最大。两年多里我看到的腐坏路径，归纳起来是三种。

**第一种：测试集渗入训练数据。** 嵌入模型的训练语料动辄几十亿 pair，而 MTEB 的很多任务源自公开数据集——MS MARCO、NQ、Quora duplicate questions。新模型的训练数据里混入这些任务的测试切分，有时是故意的，更多时候是爬虫管线无意捞进来的。结果是榜单分数逐年通胀，但你把 top 模型接到自己业务的检索场景里，提升远小于榜单差距。社区后来的应对是刻意引入训练数据公开后才构造的新任务，本质上是在和污染赛跑。

**第二种：对 prompt 模板过度敏感。** 很多人以为嵌入模型没有 prompt 问题，实际上 instruction-tuned 的嵌入模型（E5、GTE、BGE 系）对查询前缀极其敏感。`"query: "` 加不加、instruction 写成哪种措辞，在 retrieval 任务上能移动好几个点——而榜单上前十名之间的差距往往不到两个点。也就是说，**排名的一部分是模板运气，不是模型能力。** MTEB 后来强制模型作者在 model metadata 里声明 prompt 配置，就是为了把这个变量钉死。

**第三种：任务分布和真实用途脱节。** 榜单的任务混合是历史累积的结果，不是任何人的真实业务分布。有大量的英文学术检索、语义相似度任务，而实际落地的场景——中文客服 FAQ、代码检索、多轮对话记忆——权重严重不足。在均值上领先 1 分的模型，在你的场景里完全可能更差。均值是给论文用的，选型要看和你分布最接近的子任务。

## Benchmark PR 里真正吵的是什么

外面的人以为 benchmark 社区在讨论算法。其实一个新任务的 PR，review 里吵的几乎全是数据治理问题：

- 这个数据集的**license 是否允许再分发**，测试切分有没有可能已经在 CommonCrawl 里；
- 标注是众包的还是专家的，**标注者间一致性**有没有报；
- 负样本是怎么采的——随机负样本会让任务虚高，hard negative 采样又可能引入采样器自身的偏差；
- 任务规模是不是大到评一次要几个 GPU 时——**跑不起的 benchmark 没人复现，没人复现就没人信**。

最后这条引出一个 MTEB 做对了的事：评测逻辑本身是版本化的代码。`pip install mteb`，指定任务和模型，任何人都能一行命令复现榜单数字。**可复现脚本不是工程洁癖，是信任的载体**——一个数字如果只有出数的人能算出来，它就不是证据，是宣传。

这四条 review 标准——数据来源可信、标注质量可查、采样无偏、结果可复现——就是我后来做产品 eval 的验收标准。只是对象从公开数据集换成了自己的线上日志。

## 三层评测架构

在 PIN AI 我把产品评测拆成三层。分层的理由很简单：不同层的成本和检出能力差一个数量级，混在一起做，要么贵到跑不动，要么粗到看不见问题。

| 层 | 是什么 | 单次成本 | 反馈延迟 | 能抓住什么 | 抓不住什么 |
|---|---|---|---|---|---|
| L1 离线样本集 | 300–1000 条固定 case，规则判分 + LLM 判分 | 几美元 | 分钟级 | 明确回归、格式崩坏、已知坏 case 复发 | 分布外的新问题 |
| L2 影子/灰度对比 | 新旧版本对同一真实流量各跑一遍，成对比较 | 双倍推理开销 | 小时到天 | 真实分布上的相对退化 | 需要多轮交互才暴露的问题 |
| L3 在线行为信号 | 重试率、打断率、复制率、会话放弃率 | 近似免费 | 天到周 | 用户真实不满，包括你没想到测的维度 | 归因——只报警，不定位 |

三层是漏斗关系。L1 挡住廉价可挡的，L2 验证真实分布，L3 兜底所有前两层想象不到的失效模式。任何变更的路径是 L1 门禁 → L2 灰度 → L3 观察，没有例外通道——例外通道一开，整套体系一个月内失效。

L3 值得多说一句。重试率（用户对同一意图换措辞重问）和打断率（生成中途被用户掐断）是我见过最诚实的两个信号，因为用户做这些动作时没有任何讨好你的动机。满意度弹窗的数据比它们脏一个量级——愿意点弹窗的用户本身就是有偏样本。

反过来，L3 的每一次报警都应该回流成 L1 的样本：把触发重试的那条对话脱敏后沉淀进离线集。这条回路是三层架构里最容易被省略、也最不该省略的一段——没有它，L1 会以肉眼可见的速度和线上分布脱节，重演 MTEB 任务分布陈旧的老问题。

## 样本集怎么建：配比、版本、哨兵

**配比。** 我的经验值是：60% 来自线上真实失败（差评、重试、放弃会话沉淀下来的 case），25% 覆盖核心成功路径（防止只盯失败把主流程改坏），15% 对抗性构造（注入、超长输入、语种混杂、脏格式）。真实失败的占比必须过半——人造 case 测的是你想象中的用户，线上失败测的是真实用户。

**判分方式跟着题型走。** 分类、抽取、格式合规这类客观题用规则判分，确定、免费、零漂移；开放生成用 LLM 判分，但 LLM 判官本身要校准——每个月抽 50 条人工盲评，和判官打分算一次一致率，跌破 85% 就重写判分 rubric。判官模型和判分 prompt 同样要钉版本：换判官不换版本号，等于偷偷换了尺子。

**版本纪律。** 这是从 MTEB 直接搬来的：评测集必须像代码一样管理。每条样本带来源和入集日期；样本集打版本号，改动走 review；**分数永远和样本集版本一起报**。"新版 78 分" 是一句没有信息量的话，"eval-v2.3 上 78 分，对照上一版模型同集 74 分" 才是。我见过团队一边改模型一边改评测集，两个月后没人说得清分数涨的 6 分里有几分是集子变松送的。

**哨兵集，以及一次 Goodhart 式翻车。** 我们真实踩过的坑：为了压缩回答冗长度，把"简洁性"加进 LLM 判分维度并调高权重。两周里评测分涨了 4 分，L3 的追问率同步涨了 30%——模型学会了砍掉步骤中的关键前提来换简洁分，用户不得不追问一轮把前提要回来。**指标涨、体验跌，教科书级的 Goodhart：当一个度量成为目标，它就不再是好度量。**

对策是哨兵集（sentinel set）：从样本集里划出 10–15% 冻结起来，不进优化目标、不给算法和提示词工程的人看细节，只在发布前跑一次。主集分数涨、哨兵集分数平或跌，说明你在过拟合评测而不是在改进产品。哨兵集半年轮换一次——它被间接看多了也会失效，和 benchmark 被训练数据污染是同一个物理过程，只是时间尺度更短。

## 迭代日志

最后一个便宜但回报极高的习惯：每次评测驱动的变更留一条结构化记录。模板如下：

```markdown
## Iteration Log — 2025-10-14
变更:       system prompt v41 → v42（收紧工具调用前的确认话术）
动机:       L3 打断率连续两周高于基线 2.1pp
评测集版本: eval-v2.3 (harness v1.8)
L1 结果:    总分 76.2 → 78.9；tool-use 子集 +5.1；哨兵集 71.0 → 71.3（平）
L2 结果:    5% 灰度 72h，成对胜率 58/42，无子场景显著退化
决策:       全量发布
遗留:       长文档摘要子集 -0.8，未达显著性，加入下轮观察
回滚条件:   打断率 48h 内回升至基线以上即回滚 v41
```

单条看着像官僚流程，攒到第三十条就是资产：哪类变更历史上容易伤哪个子集、哨兵集多久开始漂移、L1 分数和 L3 信号的相关性有多强——这些问题只有日志能回答。**新人入职读完迭代日志，比读三份架构文档更快理解这个产品真正的约束在哪。**

## 尾声

现在评估一个 AI 团队，我只问两个问题：你们的评测集上一次删掉过时样本是什么时候？你们最近一次因为哨兵集不过而撤回发布是什么时候？两个问题都答不上来的团队，他们的 eval 不是质量体系，是给发布决策盖章用的仪式。

评测集和模型一样会过拟合、会陈旧、会说谎——区别只在于，模型骗你的时候，本该由 eval 来揭穿；而 eval 骗你的时候，没有人揭穿它。
