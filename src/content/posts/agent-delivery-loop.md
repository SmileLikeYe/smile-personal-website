过去一年，我合并了上千次 Agent 写的提交，其中大多数代码我没有逐行读过。让我敢这么干的，不是对模型的信任——恰恰是彻底的不信任。

这话听起来矛盾。展开讲。

## 一次没验证的合并

这个网站是我用 Claude Code 和 Codex 一轮一轮改出来的，从排版到路由没有一行是手写的。某天深夜，我让 Agent 把文章从首页内嵌展开改成带深链接的二级页面。它回报：「完成。深链接直达正常，浏览器后退可用。」我扫了一眼 diff，逻辑通顺，合并，睡觉。

第二天在手机上点开自己发出去的链接：直达没问题，按后退——白屏。原因很小，hash 路由只处理了前进方向，`popstate` 没接。但真正让我坐直的是另一件事：我翻回聊天记录，发现它说「后退可用」的依据，是**读了一遍自己刚写的代码**。它从头到尾没开过浏览器。

它没有撒谎。它只是给了一个「回答」。回答的验收标准是「听起来对」，交付的验收标准是「跑起来对」——这两个标准在绝大多数时候给出相同的结论,而区别只在你最不希望出错的那一小部分时刻暴露。

那晚之后，我给 Agent 和自己定了同一条规矩：一次任务只有两种合法终点。要么是可运行、已被外部信号验证的变更，要么是一份说清卡点的阻塞报告。介于两者之间的一切——「理论上应该可以」「我改好了但没跑」——一律记为失败。

后来这条规矩长成了一整套流程。可模板化的部分我整理成了站内的 agent-delivery skill，也是我开源的 agent harness [agent-chief](https://github.com/SmileLikeYe/agent-chief)（155+ stars）的设计底座。这篇不重复那份清单，讲清单是怎么被打出来的。

## 步步验证的经济学

直觉上，验证有固定成本——一次构建、三个视口的截图、一轮脚本断言，加起来一两分钟——所以攒到最后统一验一次「更省」。这个直觉错在把返工成本当成了常数。

```text
设一次任务 n 步，单步出错率 p，错误只有验证才能暴露。

最后验一次：第 k 步的错误平均要拖 (n−k) 步才被发现，
  期望返工 ∝ p · Σ(n−k) = p · n(n−1)/2   —— 随步数平方增长

每步都验：错误在第 k 步暴露就死在第 k 步，
  期望返工 ∝ p · n                        —— 线性增长
```

这个模型还是乐观的，它假设返工只是「重做」。实际上，一个第 2 步的错误被后面八个改动掩埋之后，定位本身就成了一个新任务：你面对的不再是「第 2 步错了」，而是「这十步里某处错了」，搜索空间随掩埋深度膨胀。批量验证省下的是验证的固定成本，付出的是平方级的返工加上一笔定位税。

对人类工程师，每步验证的纪律很难守住——切换成本高，而且无聊。对 Agent 这恰好是最便宜的能力：跑一次构建、拍三个视口、用十几行 Playwright 把点击、路由、剪贴板逐个断言，几十秒的事，它不会烦。所以「验证密度」是少数几个应该在 Agent 时代调得**比人类时代更高**的参数，而不是持平。

关键约束只有一条：验证信号必须来自模型之外。构建器、测试、真实浏览器里的截图——任何东西都行，唯独不能是它自己对自己代码的阅读理解。那不是验证，那是复述。

## 上下文是最稀缺的资源

第二个被重写的是信息的流动方式。Agent 的上下文窗口看着很大，实际上是它唯一不可再生的资源：塞进去的每一段 grep 输出、每一页构建日志，都在稀释它对任务本身的注意力，而且不可撤销——你没有办法让它「忘掉」刚才那两千行转储。

所以我把侦察全部外包。全库搜索、读日志、扫依赖树，这类高噪音低密度的活丢给子 agent 跑，主线程只收结论。分界线大致如下：

| 留在主线程 | 外包给子 agent |
| --- | --- |
| 任务简报与完成标准 | 全库搜索、跨文件摸排 |
| 决策与取舍的理由 | 日志与报错的原始输出 |
| 每步验证的结论与证据 | 「这个符号在哪被引用」类问题 |
| 与人的每一次确认 | 一次性的环境勘查 |

判据只有一条：主线程里的每一行，都应该是「接下来做决定所需要的东西」。侦察的过程不是，侦察的结论才是。这与带团队是同一回事——没有人让负责决策的人亲自读完全部原始日志再开会。

## 四种错，四个夜晚

失败处理是这套流程里最晚成型的部分，因为每一类错都得亲自踩过才认得出来。

**环境错。** Agent 对着 `EADDRINUSE` 改了三轮业务代码，一轮比一轮离谱。真相是上一次没杀干净的 dev server 还占着端口。从此定下第一条规则：报错先归类，凡属环境问题，禁止碰代码。

**理解错。** 我说「把文章改成二级页面」，它理解成引入一个路由框架、重写整个导航层。代码没有一行是错的，只是整个方向不是我要的。这类错任何验证都抓不住——测试全绿。解药在开工之前：先把需求写成带完成标准的简报，写不出可机器验证的完成标准，就说明我自己也不知道要什么，不许开工。

**实现错。** 报错栈明明指着它三分钟前写的函数，它的第一反应经常是「再跑一次试试」。同一条命令跑第二遍不会有不同的结果——这条对人也成立，只是 Agent 会以每分钟一次的频率原样重试。规则：读完完整错误信息之前，禁止改动。

**假设错。** 最贵的一类。全页截图拍出来一片空白，前两次修复都在改选择器、加 `waitFor`，越修越糊。真正的原因和选择器毫无关系：页面的 reveal 动画让元素在起点是透明的，截图的快门按在了动画开始之前。两次修复共享同一个错误假设——「元素没渲染出来」。规则由此而来：同一个错误连修两次不好，强制停下，把当前假设明写出来，换一个假设重新定位。在错误假设上勤奋重试，是 Agent 烧钱最快的方式。

四类错，四种处理，顺序不能乱：先排除环境，再对照简报，再读错误信息，最后才怀疑假设。跳步的代价我都替你付过了。

顺带说交付报告。任务收尾时我要求四栏：改了什么、验证了什么、**没验证什么**、已知风险。其中「没验证什么」是信任的真正来源——一份敢于写明盲区的报告，比一份全绿的报告可信得多。人如此，Agent 也如此。

## 之后，工程师做什么

把实现交给循环之后，我的时间花在两件事上，而且我认为这两件事就是这个职业接下来的形状。

一是 taste——定义「好」。完成标准清单里写什么，本身就是判断力：知道要三个视口而不是一个，知道「后退可用」必须写进验收，知道哪些边界一个字都不许动。Agent 会不折不扣地放大你的标准，也会不折不扣地放大你的模糊。

二是验证设计。为每一类变更设计出便宜、快速、来自模型之外的验证信号，正在变成一种独立的工程能力——有点像测试工程，只是对象从代码扩大成了整个交付过程。模型每几个月换一代，围绕具体模型的技巧随之作废；这两样东西是累积的，换什么模型都带得走。

我不担心 Agent 抢走工程师的工作。我担心的是工程师继续用对待搜索引擎的方式对待一个交付者——问一句，抄一段，自己收尾。那样的话，被替代确实不冤。

回答不是交付物。这条规矩管住的第一个对象，从来不是 Agent，是我。
