过去一年我大部分代码是和 Agent 一起写的。效率确实在涨，但涨幅最大的时刻不是换了更强的模型，而是我把自己的交付流程「工程化」的那几次。

## 把重复的工作沉淀成 playbook

第一次让 Agent 修 UI bug 时，我花了半小时描述背景；第十次时我意识到：这些背景说明本身就是资产。于是我开始把重复出现的工作模式写成结构化的 playbook——

- **场景**：什么情况下用这个流程（修视觉回归 / 加新接口 / 发版检查）；
- **步骤**：先看什么、改什么、怎么验证；
- **失败模式**：这类任务里 Agent 最容易犯的错，提前写进去；
- **完成标准**：什么状态才算可以交付。

playbook 的本质是把「我脑子里的隐性流程」变成显性文本。写一次，之后每个任务的启动成本趋近于零，而且流程会随着失败案例持续变好——这和团队里写 runbook 是同一件事。

## 验证信号必须来自模型之外

我给 Agent 定的规矩和给自己定的一样：**没有外部验证的改动不算完成。**

- 改代码 → 构建和测试必须过；
- 改 UI → 截图放到眼前对比，而不是听它描述「已经修好了」；
- 改交互 → 用脚本真实点一遍：点击、路由、剪贴板，每个行为断言结果。

这一条的价值在深夜最明显：当你不在场时，一个有验证闭环的 Agent 会自己发现「改动在真实渲染里根本没生效」，然后回头修；没有闭环的 Agent 会自信地报告完成。

## 人机分工：判断留给人，执行交给循环

工程化不是全自动化。我的分界线：

| 交给 Agent 循环 | 留给自己 |
| --- | --- |
| 按 playbook 执行、验证、修复 | 定义「好」的标准 |
| 穷举检查（多视口截图、全链路点击） | 取舍决策（砍什么、留什么） |
| 失败后的重试和回溯 | 对外发布前的最终判断 |

模型的能力每几个月翻一次牌，但这套流程资产是累积的。**工具会过时，工作流的工程化能力不会。**这也是我认为 AI 时代工程师真正该练的肌肉。
