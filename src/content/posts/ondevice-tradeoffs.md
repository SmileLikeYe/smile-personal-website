先说一个反直觉的结论：在端侧，模型质量是第四优先级。前三个分别是延迟、内存、电量。云端工程师听到这句话通常不信——直到他们的第一个端侧版本上线，然后被 Jetsam 杀进程的日志淹没。

我在 PIN AI 做的是常驻型 AI 硬件加 iOS 后台音频链路，模型必须在用户口袋里活着，而不是在发布会 demo 里活着。这一年里我把所有决策收敛成三本账和一套分工。下面是账本。

## 三本预算，不是三个指标

云端做容量规划，端侧做预算分配。区别在于：云端超预算是账单变贵，端侧超预算是进程死掉或者用户卸载。

**延迟预算。** 用户对首 token 的容忍度大约在 300ms 到 500ms 之间——这个数字不来自论文，来自我们把首 token 从 1.2s 压到 280ms 之后，会话完成率的变化肉眼可见。更重要的是分层：意图识别、指令改写这类任务，超过 300ms 用户就会觉得"这功能卡"；而明确的长文生成，用户愿意等 2 秒，前提是界面在流式吐字。端侧 3B 模型 250ms 出首 token，体验上碾压云端 70B 的 900ms（还没算弱网）。

**内存预算。** iOS 不跟你商量。App 的内存上限随机型浮动，经验值是：4GB RAM 的机器上，你的 App 稳定占用超过 1.5GB 就进入 Jetsam 的射程，超过 2GB 基本是随时会死。而一个 7B fp16 模型光权重就 14GB——这句话本身就宣判了大部分"把云端模型搬下来"的方案。真实的可用窗口是：模型权重 + KV cache + App 自身 + 系统波动余量，全部塞进大约 1.2GB。这意味着 3B int4 是常态，7B 是奢侈品。

**电量预算。** 最容易被忽略，因为 demo 阶段没人插着功耗计。实测经验：持续跑 CPU/GPU 推理，iPhone 每小时掉电 15% 到 25%，机身发热到用户能感知的程度大约只要 8 分钟。用户不会说"这个 App 推理太频繁"，他会在设置里看到你排在电池榜第一，然后卸载。ANE（神经引擎）的能效比 GPU 高一个量级，所以能走 Core ML + ANE 的路径就不要碰 GPU——哪怕 ANE 的算子支持列表会逼你改模型结构。

三本账的共同点：超支的惩罚都不是"变慢"，而是"被杀"或"被卸"。这就是为什么质量排第四。

## 量化：唯一免费午餐在哪停下

量化是端侧最大的杠杆，但杠杆有支点。以 3B 模型为例：

| 精度 | 权重体积 | 相对速度 | 质量表现 |
|------|---------|---------|---------|
| fp16 | ~6 GB | 1x | 基线 |
| int8 | ~3 GB | 1.5–2x | 几乎无损，盲测分不出 |
| int4 | ~1.6 GB | 2–3x | 出现质量悬崖：长逻辑链断裂、多语言退化、数字计算出错 |

int8 是真正的免费午餐。int4 不是——它是一笔交易，能不能做取决于任务类型：

| 任务类型 | int4 可接受？ | 原因 |
|---------|-------------|------|
| 分类 / 意图识别 | 可以 | 输出空间小，量化噪声被 argmax 吸收 |
| 抽取 / 改写 | 可以 | 有原文锚定，模型不需要"自己想" |
| 短回复生成 | 勉强 | 需要针对性评测，别信通用 benchmark |
| 长推理 / 数学 / 代码 | 不行 | 误差在长链条上累积，悬崖就在这里 |

我的实操规则：端侧只部署"输出被输入强约束"的任务。让 int4 模型自由发挥，等于把质量悬崖直接推给用户。

## 混合架构：分工表比模型选型重要

端侧 vs 云端不是二选一，是流水线分工。我们的划分：

| 环节 | 位置 | 理由 |
|------|------|------|
| 唤醒/触发过滤 | 端侧 | 每天触发上万次，走云端在延迟和成本上都是自杀 |
| VAD 式预过滤 | 端侧 | 把 24 小时音频流削减到 <5% 有效片段，是整条链路的成本闸门 |
| 隐私敏感抽取 | 端侧 | 原始音频/健康/财务数据不出设备，出去的只有结构化摘要 |
| 轻量意图理解 | 端侧 | 300ms 预算内唯一的选择 |
| 重推理/长生成 | 云端 | 质量差距太大，端侧模型追不上，也不该追 |

注意第三行：隐私不是把所有东西留在端侧，而是把"原始数据到脱敏特征"这一步留在端侧。这比"全端侧"的叙事诚实得多，也可实现得多。

离线时的降级不能是 if-else 补丁，得是路由器的一等公民：

```swift
func route(_ task: InferenceTask) -> Executor {
    // 端侧专属任务：无条件本地，云端挂了也不影响
    if task.tier == .edgeOnly { return .local }

    // 云端任务：网络、热压、电量三重闸门
    let cloudViable = network.isUsable
        && thermalState < .serious
        && battery.level > 0.15

    if cloudViable { return .cloud(fallback: .local) }

    // 降级：能力残缺但功能存活，并且明确告诉用户
    return task.hasLocalPath
        ? .localDegraded(banner: .offlineMode)
        : .queued(retryWhenOnline: true)
}
```

关键有两个。一是降级要"诚实"——离线时给一个明显更短、更保守的回答并挂上离线标识，好过给一个假装正常的烂回答。二是热和电也是路由信号：机身过热时把任务推去云端，反而比本地硬扛更省用户的电。

## iOS 的账单：三条用真机换来的教训

**后台不是你的。** iOS 给后台任务的标准窗口是 30 秒量级，`BGProcessingTask` 什么时候调度、调不调度，系统说了算。我们做后台音频，靠的是 audio session 这条合法通道续命，但即便如此，后台内存红线比前台低得多——前台能活的模型，后台加载就是闪退。结论：后台只做采集和最轻量的过滤，任何推理等回前台或推到云端。

**热节流是斜坡不是开关。** 机身升温后系统悄悄降频，推理延迟在 10 分钟内从 250ms 爬到 700ms，没有任何回调告诉你为什么。等 `thermalState` 报 `.serious` 时用户已经体验了几分钟的卡顿。我们后来直接监控滑动窗口内的推理延迟漂移，超阈值就主动降档（缩上下文、切小模型、限制触发频率），不等系统通知。

**Demo 机和舰队是两个世界。** 办公室里是插着电、刚重启、机身冰凉的 iPhone 16 Pro；线上舰队的中位数是一台三年前的机器，RAM 少一半，没有为你的模型优化过的 ANE，电池健康 85%，后台还挂着微信。我们在 15 Pro 上首 token 280ms 的配置,同一份代码在 iPhone 12 上是 1.4s 加一次热节流。做过跨 iOS/Android/Web 的企业级发布之后我对这件事有肌肉记忆：适配矩阵的下限决定产品评分，上限只决定发布会效果。现在我们的性能验收基线是"三年前的中端机、电量 30%、后台有音乐"，旗舰机数据只准出现在 PPT 里。

## 收敛

端侧不是小一号的云端，它是另一种工程文化：预算先立、降级先行、验收看舰队最差的那台机器。这套纪律反过来带回云端，你会发现自己写的系统突然变便宜了。

云端工程师优化的是模型的上限，端侧工程师守的是它的下限——而用户的评价，从来都写在下限上。
