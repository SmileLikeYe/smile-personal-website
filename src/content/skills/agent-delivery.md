大部分人用 Agent 的方式是"提问 → 拿到回答 → 自己动手"。这份 skill 的目标是把 Agent 用成一个**交付者**：任务结束时得到的是一个能跑、被验证过的产品面，而不是一段需要人工消化的文字。

## 核心原则

**回答不是交付物。** 一次 Agent 任务的终点只有两种：可运行且验证过的变更，或一份明确说明"卡在哪、需要什么"的阻塞报告。介于两者之间的一切（"你可以试试……"）都算失败。

## 循环（每个任务都走完整五步）

### 1. 计划（Plan）

- 把任务改写成**可验证的完成标准**："改好登录页" → "登录页在 1440/390 两个视口下渲染正确，账号密码错误时有行内报错"
- 列出会动到的文件和不确定的点；不确定的点先查证，不要边写边猜

### 2. 工具（Act with tools）

- 能用工具核实的绝不凭记忆：查文件、跑命令、搜代码
- 大范围探索交给子任务/子 Agent，主线程只保留结论

### 3. 检查（Inspect）

- 每次修改后立刻构建 / 跑测试 / 截图，不要攒到最后
- UI 变更必须截图：桌面 + 移动两个视口起步；交互变更必须脚本化点一遍

### 4. 修复（Fix）

- 出错时先读完整错误信息再动手，禁止"再跑一次试试"
- 同一个错误修两次没好，就停下来换假设：是不是改错了层？

### 5. 验证（Verify）

- 对照第 1 步的完成标准逐条打勾
- 交付时说清楚：改了什么、验证了什么、什么没验证、已知风险

## 反模式（看到就纠正）

- 交付一段代码但没跑过 —— 等于没交付
- 测试失败但说"应该没问题" —— 报告事实，不粉饰
- 把"下一步你可以……"留给用户 —— 下一步是你的，现在就做
- 在一个坑里连续尝试 3 次以上 —— 停，重新计划
