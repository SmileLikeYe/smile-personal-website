大部分人用 Agent 的方式是"提问 → 拿到回答 → 自己动手收尾"。这份 skill 把 Agent 用成**交付者**：任务结束时得到的是一个能跑、被验证过的产品面。它不是理念，是一套我在真实项目（包括本站的多轮改版）里反复运转的操作系统，每一步都有模板和判据。

## 核心不变量

**回答不是交付物。** 一次任务只有两种合法终点：

1. 可运行、已验证的变更 + 交付报告
2. 一份阻塞报告：卡在哪、试过什么、需要对方提供什么

介于两者之间的一切——"你可以试试"、"理论上应该"、"我改好了但没跑"——都记为失败。

## 第一步：任务简报（写给 Agent 的 PRD）

任务开始前先把需求改写成简报，四个字段缺一不可：

```text
目标     一句话说清改完之后世界有什么不同
完成标准 可机器验证的清单（构建通过 / 截图正确 / 测试绿 / 链接可点）
边界     什么不许动（不许改公共 API / 不许升级依赖 / 不许动别人的提交）
环境     分支策略、部署方式、验证命令
```

写不出"完成标准"的任务不要开工——那说明你自己也不知道要什么，Agent 只会放大这种模糊。

**实例**（本站某轮改版的简报）：目标=文章改为二级页面；标准=打开滚到顶、浏览器后退可用、深链接直达、三视口截图正常；边界=不动训练面板；环境=worktree 分支，`npm run build` + Playwright 脚本验证。

## 第二步：隔离与核实

- **物理隔离**：在 git worktree / 独立分支里干活，主干永远可发布。Agent 犯错的成本必须被架构限制住，而不是靠它"小心"
- **凭证先行**：能用工具核实的绝不凭记忆——读文件确认现状、跑命令确认版本、搜代码确认没有第二处调用点
- **大范围侦察外包**：grep 全库、读日志这类高噪音工作丢给子任务，主线程只保留结论。上下文是 Agent 最稀缺的资源，别让它被文件转储淹死

## 第三步：步步验证（不是最后验证）

每完成一个可验证的最小单元，立刻验证，绝不攒批：

- 改样式 → 构建 + **三视口截图**（1440/768/390）
- 改交互 → **脚本化断言**（点击、路由、剪贴板，Playwright 十几行就够）
- 改数据流 → 单元测试或最小复现脚本

两条来自血泪的截图脚本纪律：程序化滚动必须 `behavior:"instant"`（页面开了 smooth scroll 会滚不到位）；全页截图前先注入 `animation: none` 冻结动画（否则 reveal 元素拍出来全是空白）。

## 第四步：失败分类学

出错时先归类再动手，四类错误四种处理：

| 类型 | 特征 | 处理 |
| --- | --- | --- |
| 环境错 | 命令不存在 / 端口占用 / 权限 | 修环境，不动代码 |
| 理解错 | 改对了地方但需求理解偏了 | 回到简报重读，找人确认 |
| 实现错 | 报错栈指向自己刚写的代码 | 读完整错误信息再改，禁止"再跑一次试试" |
| 假设错 | 同一个坑修两次都没好 | 停。写下当前假设，换一个假设重新定位 |

第四类最贵：Agent（和人）在错误假设上的重试是纯烧钱。规则是**同一错误连续两次修复失败即强制换假设**。

## 第五步：交付报告

```text
改了什么   文件级清单 + 一句话意图
验证了什么 对照完成标准逐条打勾，附证据（截图/测试输出）
没验证什么 明确列出，附原因
已知风险   哪里可能炸、炸了怎么回滚
```

"没验证什么"这一栏是信任的来源——一份敢写盲区的报告，比一份全绿的报告更可信。

## 反模式速查

- 交付没跑过的代码 —— 等于没交付
- 测试红着说"应该没问题" —— 报告事实，不粉饰
- 把"下一步你可以…"留给用户 —— 下一步是你的，现在就做
- 在最后一刻做一次"大验证" —— 攒批验证 = 攒批返工
- 上下文里塞满原始日志 —— 侦察外包，结论进主线
