# Agent 交付循环（Smile 的工作流）

核心不变量：**回答不是交付物。** 一次任务只有两种合法终点：
1. 可运行、已验证的变更 + 交付报告
2. 阻塞报告：卡在哪、试过什么、需要对方提供什么

介于两者之间的一切（"你可以试试"、"理论上应该"、"改好了但没跑"）都算失败。

## 第一步：任务简报

开工前把需求改写成简报，四个字段缺一不可：

```text
目标     一句话说清改完之后世界有什么不同
完成标准 可机器验证的清单（构建通过 / 截图正确 / 测试绿 / 链接可点）
边界     什么不许动（公共 API / 依赖版本 / 他人的提交）
环境     分支策略、部署方式、验证命令
```

写不出完成标准的任务不要开工——先向用户澄清。

## 第二步：隔离与核实

- 在 git worktree / 独立分支里干活，主干永远可发布
- 能用工具核实的绝不凭记忆：读文件确认现状、跑命令确认版本、搜代码确认调用点
- 大范围侦察（全库 grep、日志翻查）交给子任务，主线程只保留结论——上下文是最稀缺资源

## 第三步：步步验证（不攒批）

每完成一个可验证的最小单元立刻验证：

- 改样式 → 构建 + 三视口截图（1440 / 768 / 390）
- 改交互 → 脚本化断言（点击、路由、剪贴板）
- 改数据流 → 单元测试或最小复现脚本

截图脚本纪律：程序化滚动用 `behavior:"instant"`；全页截图前注入 `animation: none` 冻结动画。

## 第四步：失败分类，再动手

| 类型 | 特征 | 处理 |
| --- | --- | --- |
| 环境错 | 命令不存在 / 端口占用 / 权限 | 修环境，不动代码 |
| 理解错 | 改对了地方但需求理解偏了 | 回简报重读，找人确认 |
| 实现错 | 报错栈指向刚写的代码 | 读完整错误再改，禁止盲目重跑 |
| 假设错 | 同一个坑修两次没好 | 停，写下当前假设，换假设重定位 |

**同一错误连续两次修复失败，强制换假设。**

## 第五步：交付报告

```text
改了什么   文件级清单 + 一句话意图
验证了什么 对照完成标准逐条打勾，附证据
没验证什么 明确列出 + 原因
已知风险   哪里可能炸、怎么回滚
```

"没验证什么"必须写——敢写盲区的报告才可信。

## 反模式（看到即纠正）

- 交付没跑过的代码
- 测试红着说"应该没问题"
- 把"下一步你可以…"留给用户——下一步是你的，现在就做
- 攒到最后做一次大验证
- 把原始日志塞进主线上下文
